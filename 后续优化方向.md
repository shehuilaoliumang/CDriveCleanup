# 后续优化方向

本项目目前已完成主流程修复与界面可用性改善，建议在不改变核心主流程的前提下按以下方向继续迭代。

## 1. 定时清理能力升级
- 已支持：
  - 每天
  - 每周
  - 每 10 天
  - 每 15 天（半月）
- 建议优化：
  - 在设置中加入“每 30 天”快捷按钮
  - 显示下次执行时间（`YYYY-MM-DD HH:MM`）
  - 增加任务执行失败重试和失败记录提示

## 2. 清理执行与数据安全
- 保持“预览 -> 确认 -> 执行”流程不变，避免直接误删。
- 建议将备份失败的文件记录为“未清理”项并加入失败计数，避免无备份情况下误删高风险文件。
- 增加“清理前总览”与“清理后执行摘要”更细粒度展示（例如失败文件路径上限和原因）。

## 3. 扫描与视图性能
- 大目录扫描时建议增加扫描队列/分页策略，防止 UI 卡顿。
- 建议在列表显示中加入“按文件大小、按路径关键字”的二级过滤，降低一次性渲染负载。
- 已有图表与日志建议统一使用 `ttk.Treeview` 的列宽自适应策略（当前已具备手工滚动条增强）。

## 4. 打包与发布（EXE）
- 当前主流方案（单文件/单目录）可继续使用，不影响模块拆分。
- 增加一个固定的发布流程文档：
  1. 安装依赖
  2. 清理临时构建产物
  3. 执行打包
  4. 垃圾文件与体积检查
- 打包失败时建议输出 `--log-level INFO` 与 `--clean` 重试流程。

## 5. 维护与质量
- 引入最小化回归测试（至少 5 条场景）：
  - 备份开关下的清理行为
  - 模拟模式
  - 定时执行间隔解析
  - 日志文件读写
  - 恢复备份功能
- 定期检查 `tmp/`、`dist/`、`build/`、`__pycache__/` 的清理策略，避免本地占用堆积。
